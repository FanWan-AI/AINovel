## 1. 文档定位

本文档是本项目该升级主题的产品与工程统一规格，目标是直接指导前后端并行开发、测试验收与灰度发布。

## 2. 目标与非目标

## 2.1 目标

1. 用户在 AI助手页面通过对话即可完成“选书-执行-复盘”的创作闭环。
2. 用户不再依赖右侧侧滑助手完成复杂任务。
3. 设置中心只放系统级治理项，不与书籍页动作重复。
4. 题材设置集中管理，并明确每个字段是否真实生效。
5. 工程上支持分阶段迁移，避免一次性大爆炸改造。
6. 建立可量化增长目标，确保升级不仅可用，而且能持续提升留存与口碑。

## 2.2 非目标

1. 本期不做模型训练与提示词平台化改造。
2. 本期不改写核心写作算法，仅做编排与体验升级。
3. 本期不覆盖移动端壳层适配细节。

## 3. 用户与关键场景

## 3.1 用户分层

1. 新手作者：希望一句话触发写作，不关心复杂参数。
2. 进阶作者：希望在聊天中可控地指定章节、约束、修订策略。
3. 运营人员：希望通过聊天快速跑审计、雷达、复盘流程。

## 3.2 关键任务场景

### 场景 A：写下一章

1. 用户进入 AI助手页面。
2. 选择书籍范围为单本书。
3. 输入“写下一章，保持债务线冲突，章末留钩子”。
4. 助手生成执行计划并要求确认。
5. 用户确认后执行，页面显示实时进度。
6. 完成后给出结果摘要与下一步建议。

### 场景 B：审计指定章节并精修

1. 用户输入“审计第 14 章”。
2. 助手若未识别书籍则要求先选书。
3. 审计结果返回后，助手给出“是否发起 spot-fix/polish”的建议卡。
4. 用户点击建议直接进入下一步执行。

### 场景 C：批量运营任务

1. 用户选择“全部活跃书”。
2. 输入“跑一次市场雷达并生成简报”。
3. 助手执行并按书输出结果摘要。

## 4. 当前问题与根因

## 4.1 AI助手能力受限

问题：

1. 右侧面板过窄，无法承载复杂确认和结果卡片。
2. 会话上下文与书籍上下文绑定弱，误执行风险高。
3. 当前流程偏“命令输入”，缺少任务编排与恢复。

根因：

1. 入口与页面结构设计为辅助工具，而非主任务面板。
2. 缺少统一任务状态机与执行契约。

## 4.2 设置职责混乱

问题：

1. 配置、题材、书籍级操作分散，用户心智割裂。
2. 设置与书籍页有重复项，造成“改哪里都像对”的困惑。

根因：

1. 缺少系统级设置边界定义。
2. 缺少“全局策略 vs 单书动作”优先级规则。

## 6. AI助手主页面详细设计

## 6.1 页面定位

AI助手为主内容区页面（中间主视图），不再依赖右侧 ChatPanel。

页面职责：

1. 对话理解与任务规划。
2. 任务参数确认与执行。
3. 执行结果复盘与下一步建议。

## 6.2 页面布局与组件

采用三段式：

1. 顶部上下文条 ContextBar。
2. 中间对话与任务流 MessageStream。
3. 底部输入与快捷动作 Composer。

### 6.2.1 ContextBar

字段：

1. 书籍范围：single | multi | all-active。
2. 当前目标书籍列表。
3. 当前会话模式：normal | guided | batch。
4. 最近执行状态：idle | running | failed | done。

交互规则：

1. 书籍级动作若未选范围，触发阻断提示。
2. 切换范围时不清空历史消息，但会重置未确认任务。

### 6.2.2 MessageStream

消息类型：

1. user-text：用户消息。
2. assistant-plan：助手计划。
3. param-card：参数确认卡。
4. run-event：执行事件流。
5. result-card：结果摘要卡。
6. recovery-card：失败恢复卡。

呈现策略：

1. 计划与结果卡可折叠。
2. run-event 支持阶段聚合，避免刷屏。
3. 错误卡必须包含“重试/改参数/取消”按钮。

### 6.2.3 Composer

能力：

1. 自然语言输入。
2. 斜杠命令自动补全。
3. 快捷动作芯片。

首期快捷动作：

1. 写下一章。
2. 审计章节。
3. 精修章节。
4. 市场雷达。
5. 同步事实（resync）。

---

## 6.3 聊天即创作的智能体能力模型

## 6.3.1 分层能力

### A. 意图层

1. 识别用户意图与动作类型。
2. 抽取参数（chapter、mode、brief、scope）。

### B. 规划层

1. 生成可执行计划。
2. 校验参数完整性与风险。

### C. 执行层

1. 调用既有 API。
2. 订阅执行事件并回传。

### D. 恢复层

1. 失败分类。
2. 提供修复路径。

## 6.3.2 动作映射

首期动作映射：

1. write-next -> /api/books/:id/write-next
2. audit -> /api/books/:id/audit/:chapter
3. revise -> /api/books/:id/revise/:chapter
4. rewrite -> /api/books/:id/rewrite/:chapter
5. resync -> /api/books/:id/resync/:chapter
6. radar -> /api/radar/scan

参考：

1. [packages/studio/src/api/server.ts](../packages/studio/src/api/server.ts)

## 6.3.3 命令语法（对话辅助）

支持自然语言为主，命令为辅。建议保留以下内部语法用于 parser：

1. /write next book=<id> brief=<text>
2. /audit book=<id> chapter=<n>
3. /revise book=<id> chapter=<n> mode=<spot-fix|polish|rewrite|rework|anti-detect>
4. /radar

注意：用户可不用命令，系统自动转为内部结构化意图。

---

## 6.4 任务状态机

统一状态机：

1. draft：已识别意图，未确认。
2. awaiting-confirm：等待用户确认。
3. running：执行中。
4. succeeded：执行成功。
5. failed：执行失败。
6. cancelled：用户取消。

状态转换规则：

1. draft -> awaiting-confirm：参数完整且涉及执行。
2. awaiting-confirm -> running：用户确认。
3. running -> succeeded/failed：执行结束。
4. awaiting-confirm -> cancelled：用户取消。
5. failed -> awaiting-confirm：用户选择“修参重试”。

---

## 6.5 防错与安全约束

1. 未选书籍不能执行书籍级动作。
2. rewrite、delete、rollback 等高风险动作必须二次确认。
3. 同一本书的互斥任务并发时给出队列提示。
4. 所有执行消息必须回显目标对象（bookId/chapter/runId）。
5. 用户可一键中止长任务并保留日志。

## 7. 设置中心重构详细设计

## 7.1 设计原则

1. 去重：不放书籍页面已有单次动作。
2. 分层：只放系统级、跨书级治理项。
3. 可解释：每个高风险项有说明与默认值。
4. 可回滚：设置变更可恢复到默认。

## 7.2 设置分页结构

保留 5 个顶级分页：

1. 语言与地区
2. LLM Provider 设置
3. 题材设置
4. 外观与体验
5. 写作偏好

## 7.3 各分页字段级设计

### 7.3.1 语言与地区

字段：

1. uiLanguage：zh | en，默认 zh。
2. timeFormat：24h | 12h，默认 24h。
3. numberFormat：cn | en，默认 cn。

交互：

1. 语言切换后显示即时预览。
2. 保存后全局生效并提示刷新可选。

### 7.3.2 LLM Provider 设置

迁移自原 ConfigView：

1. provider
2. model
3. baseUrl
4. stream
5. temperature
6. maxTokens
7. modelOverrides（agent 维度）

约束：

1. temperature 范围校验。
2. maxTokens 数值校验。
3. 保存失败需保留草稿。

### 7.3.3 题材设置

迁移自原 GenreManager：

1. 题材列表（内置+项目）。
2. 题材详情。
3. copy/create/edit/delete。
4. 生效说明卡。

生效说明卡内容：

1. 当前已生效字段。
2. 预留字段。
3. 影响链路说明。

### 7.3.4 外观与体验

字段：

1. themeMode：light | dark | system。
2. motionLevel：full | reduced。
3. density：comfortable | compact。
4. notificationLevel：all | important-only。

### 7.3.5 写作偏好（系统治理层）

注意：避免与书籍页重复。本分页不放“默认导出格式、默认章节动作”等单书操作项。

字段建议：

1. stylePreset：balanced | vivid | concise。
2. auditStrictness：relaxed | standard | strict。
3. antiAIGuard：low | medium | high。
4. postWritePolicy：readability-first | consistency-first | balanced。

优先级规则：

1. 单书显式参数优先于全局偏好。
2. 全局偏好作为默认基线，仅在单书未指定时生效。

## 8. 题材功能解释（产品可读版）

## 8.1 题材的真实作用

题材不是“分类标签”，而是“规则包”，影响写作提示、修订约束和后验校验。

当前已验证链路：

1. fatigueWords -> 后写作校验。
2. pacingRule -> 写作前检查文本。
3. numericalSystem -> 修订阶段账本更新要求。
4. powerScaling -> 写作检查项注入。

参考：

1. [packages/core/src/agents/rules-reader.ts](../packages/core/src/agents/rules-reader.ts)
2. [packages/core/src/agents/post-write-validator.ts](../packages/core/src/agents/post-write-validator.ts)
3. [packages/core/src/agents/en-prompt-sections.ts](../packages/core/src/agents/en-prompt-sections.ts)
4. [packages/core/src/agents/reviser.ts](../packages/core/src/agents/reviser.ts)

## 8.2 字段分级建议

建议在 UI 中显示字段状态：

1. 已生效：fatigueWords、pacingRule、numericalSystem、powerScaling。
2. 预留/弱生效：chapterTypes、auditDimensions（逐步接入）。

## 12. 验收标准

## 12.1 功能验收

1. AI助手在中间主视图可独立完成核心动作。
2. 用户可选书籍范围并获得参数确认。
3. 执行中有阶段反馈，执行后有结构化结果卡。
4. 设置中心五分页可用且不重复。
5. 题材设置可在 settings 中完成全生命周期管理。

## 12.2 体验验收

1. 主路径操作不超过 3 次确认点击。
2. 错误提示可操作，不只给报错文本。
3. 页面在桌面宽度下无横向裁切。

## 12.3 稳定性验收

1. 关键路径无阻断性前端报错。
2. 执行失败不丢会话上下文。
3. 重试与取消行为一致可预期。

## 12.4 指标验收（上线后 4 周观测）

1. 新用户首次完成“写下一章”成功率 >= 75%。
2. AI助手会话到实际动作执行转化率 >= 45%。
3. 助手任务失败后二次重试成功率 >= 60%。
4. 设置页跳出率较旧入口模式下降 >= 20%。
5. 周活跃创作者中，使用 AI助手主页面的占比 >= 65%。

## 14. 风险与缓解

## 17. 爆款潜力评估与增强策略

结论：当前方案“可用且可落地”，但要达到“能火”，必须把产品从“功能可执行”升级为“结果可感知、成长可持续、分享可传播”。

## 17.1 当前方案的强项

1. 主入口统一，降低认知负担。
2. 任务状态机完整，失败可恢复。
3. 设置边界清晰，减少重复配置。

## 17.2 当前方案的短板

1. 缺少明确的增长飞轮，用户完成任务后没有强驱动回流。
2. 缺少“成就反馈”，用户感受不到持续进步。
3. 缺少传播触发点，难以形成外部口碑扩散。

## 17.3 增长飞轮设计

定义飞轮：完成创作任务 -> 获得可见收益 -> 形成下一步建议 -> 一键继续执行 -> 产出更多成果 -> 提升信任与活跃。

落地机制：

1. 结果卡增加“收益摘要”：本次提升点、风险下降点、建议下一步。
2. 每次完成后给出“下一步一键动作”，减少思考成本。
3. 在运行中心增加“本周完成里程碑”，强化累计感。
4. 支持导出“创作日报摘要”，便于用户复盘与分享。

## 18. 可用性与人性化增强规范

## 18.1 首次使用体验（First 5 Minutes）

1. 首次进入 assistant 页面，展示 3 个可直接点击的示例目标。
2. 若用户未选书籍，输入框上方显示轻提示而非硬报错。
3. 首次成功执行后弹出轻量引导，解释“如何继续下一步”。

## 18.2 对话体验人性化

1. 助手回复采用“先结果、后解释”的结构，降低阅读负担。
2. 失败信息必须包含“原因 + 可操作修复按钮 + 预计耗时”。
3. 长任务执行时显示阶段进度与剩余预估，减少焦虑。
4. 高风险动作确认文案采用自然语言，不使用仅技术术语。

## 18.3 可访问性与包容性

1. 所有核心操作支持键盘完成。
2. 关键按钮与状态提示满足对比度要求。
3. 动画可降级，motionLevel=reduced 时禁用干扰动画。
4. 错误与成功提示不只依赖颜色，必须有图标和文本。

## 18.4 信任与可控感

1. 每次执行前明确展示目标对象（书籍、章节、动作）。
2. 每次执行后提供“查看依据”入口，说明为何给出该建议。
3. 用户可查看最近 20 次助手动作记录并支持撤销入口（可撤销范围内）。
4. 对模型输出不确定内容，标记“建议级别”避免误导。

## 19. 商业化与口碑传播设计

## 19.1 用户价值主张

1. 不是“聊天工具”，而是“可执行的创作操作系统”。
2. 核心卖点是“少决策成本 + 高可控 + 有结果闭环”。

## 19.2 传播触发器

1. 创作周报可一键生成分享版摘要。
2. 里程碑卡片可复制为文本，便于社媒发布。
3. 助手给出的关键改进点支持一键复制到创作笔记。

## 19.3 反同质化要求

1. 禁止把助手页面做成普通聊天窗口，必须保留任务卡与结果卡双轨结构。
2. 结果卡必须包含“可执行下一步”，而非仅给建议。
3. 设置页必须保持“系统治理”边界，避免回到历史重复配置问题。

## 20. 发布与运营策略

## 20.1 分阶段发布

1. 内测阶段：10-20 位核心作者，重点验证任务闭环与错误恢复。
2. 小流量阶段：10% 用户，观察转化率与失败率指标。
3. 全量阶段：满足指标门槛后全量，并保留 1 个版本回滚开关。

## 20.2 发布门槛（Go/No-Go）

1. 阻断性缺陷为 0。
2. 首次任务成功率达到目标区间。
3. 关键路径平均响应耗时可接受：
	1. 计划生成 <= 3 秒。
	2. 动作触发反馈首包 <= 2 秒。
	3. 非长任务完成反馈 <= 15 秒。

## 20.3 上线后运营动作

1. 每周分析失败任务 Top 5 原因并滚动修复。
2. 每两周更新一次示例提示词与快捷动作文案。
3. 每月评估题材生效字段覆盖率，逐步提升“已生效字段”占比。
