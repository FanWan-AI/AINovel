# 维护治理与持续迭代规范

## 1. 目标

确保该平台能长期演进，避免“功能增长后不可维护”。

## 2. 分支与发布规范

分支建议：
- `main`：稳定主线
- `release/*`：发布分支
- `feature/*`：功能分支
- `fix/*`：缺陷修复

提交规范建议：
- `feat:`
- `fix:`
- `refactor:`
- `docs:`
- `test:`

## 3. 需求流程

1. PRD 评审（产品 + 研发）
2. 技术评审（架构 + 接口 + 风险）
3. 任务拆分（前后端 + 测试）
4. 开发与联调
5. 验收与灰度
6. 全量发布

## 4. 文档维护机制

要求：
- 功能 PR 必须同步更新对应文档节。
- 契约变更必须更新 `06-数据模型与契约规范.md`。
- 发布后更新 Roadmap 与已完成里程碑。

## 5. 代码治理

- 保持前后端契约集中管理，避免散落类型定义。
- 严禁在 UI 层写业务规则，规则下沉到服务层。
- 所有新接口必须带错误码。

## 6. 技术债管理

每个迭代至少处理 10%-15% 研发容量用于技术债：
- 重复逻辑收敛
- 测试缺口补齐
- 性能热点治理

## 7. 版本策略

建议语义化版本：
- `MAJOR`：接口不兼容
- `MINOR`：新增功能兼容
- `PATCH`：缺陷修复

## 8. 决策记录（ADR）

建议新增目录：`DevDocs/adr/`

记录以下关键决策：
- 为什么采用双模式而非单模式
- 为什么继续使用文件系统而非数据库
- brief 与基础文件的关系定义

## 9. 角色协作建议

- 产品：定义体验与验收
- 前端：交互与状态管理
- 后端：接口与流程编排
- 测试：主路径回归与自动化
- 运维：监控、故障响应、发布保障

## 10. 评审清单

合入前必须确认：
1. 主流程可跑通。
2. 失败路径可恢复。
3. 埋点与日志完整。
4. 文档已更新。
5. 测试通过。

## 11. Agent 协作治理（新增）

- 任务拆分粒度：
  - 每个任务最多改 3-5 个相关文件。
  - 必须声明“本任务不触碰范围”。

- 合并治理：
  - Agent 提交必须包含变更摘要与测试证据。
  - 高风险目录（鉴权、同步、删除）强制人工审批。

- 持续改进：
  - 每周复盘 Agent 任务成功率与返工率。
  - 对高返工场景补充模板和自动检查器。
