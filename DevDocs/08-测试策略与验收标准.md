# 测试策略与验收标准

## 1. 测试目标

- 核心创建流程可稳定完成。
- 双模式流程行为一致且可预测。
- 章节连续性提示逻辑可复现。

## 2. 测试分层

## 2.1 单元测试

后端：
- brief schema 校验
- brief normalize 逻辑
- create-flow 状态机转移
- 错误码映射

前端：
- 模式切换组件
- 摘要编辑组件
- 补问卡片交互
- 创建按钮可用性条件

## 2.2 集成测试

- `POST /normalize -> /confirm -> /create-status` 全链路
- 创建失败重试
- 同名书籍冲突 409
- brief 持久化文件写入验证

## 2.3 端到端测试（E2E）

场景 S1：新手快速创建
1. 选择快速模式
2. 输入自然语言
3. 确认摘要
4. 创建成功并跳转书籍详情

场景 S2：专业模式创建
1. 完成四步输入
2. 预览基础文件
3. 确认创建

场景 S3：异常恢复
1. 模型异常导致 normalize 失败
2. 用户手动编辑并继续创建

## 3. 回归测试范围

每次发布必须回归：
- 现有 `BookCreate` 基础流程（旧流程）
- 章节写作接口（write/draft/revise/audit）
- 日志与 SSE 不可中断

## 4. 数据测试集建议

准备 6 组标准输入：
1. 极简一句话输入
2. 超长详细输入
3. 缺失主角信息
4. 缺失结局方向
5. 中英混合输入
6. 含敏感词输入

用于验证 normalize 稳定性与补问逻辑。

## 5. 验收标准（MVP）

功能验收：
1. 快速模式可完成建书。
2. 专业模式可完成建书。
3. 摘要确认页可编辑并生效。
4. 创建失败有明确错误提示。

质量验收：
1. 核心接口成功率 >= 98%（测试环境）。
2. 前端主流程无阻断级 bug（P0/P1）。
3. E2E 主路径通过率 100%。

## 6. 自动化建议

- CI 必跑：
  - `pnpm -r test`
  - `pnpm -r typecheck`
- 针对新接口补充 API 契约快照测试。
- nightly 跑一次创建流程 E2E。

## 7. 缺陷分级

- P0：阻断创建/数据丢失
- P1：主流程严重错误但可绕过
- P2：体验缺陷/提示不清
- P3：样式与次要问题

发布门禁：
- 不允许存在 P0/P1 未修复缺陷。

## 8. 跨端与上架测试（新增）

- iOS：
  - TestFlight 核心路径全通过
  - 隐私权限与账号删除流程可用

- Android：
  - 主流机型兼容性
  - 前后台切换恢复

- Windows/macOS：
  - 安装升级卸载
  - 本地缓存与同步恢复

## 9. Agent 回归测试（新增）

- 每个任务由 Agent 自动执行：
  - 类型检查
  - 单元测试
  - 最小 E2E
  - 文档链接检查

## 10. 可控写下一章 E2E 验收流程

### 10.1 流程概述

> 覆盖链路：填干预 → 预览 plan → 写下一章 → SSE 完成

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 编辑 `story/current_focus.md` 或在 Studio 触发 draft with context | 干预内容写入 |
| 2 | `POST /api/books/:id/draft` + body `{ "context": "..." }` | `{ "status": "drafting", "bookId": "..." }` 立即返回；SSE `draft:start` 到达 |
| 3 | 等待 SSE `draft:complete` 事件 | `{ "bookId": "...", "chapterNumber": N, "title": "...", "wordCount": N }` |
| 4 | `POST /api/books/:id/write-next` | `{ "status": "writing", "bookId": "..." }` 立即返回；SSE `write:start` 到达 |
| 5 | 等待 SSE `write:complete` 事件 | `{ "bookId": "...", "chapterNumber": N, "status": "ready-for-review", ... }` |
| 6 | `GET /api/books/:id` 验证章节列表 | 新章节出现，`status` 为 `"ready-for-review"` |

### 10.2 接口字段速查

**POST `/api/books/:id/write-next`**
```json
// 请求体（可选）
{ "wordCount": 3000 }
// 响应
{ "status": "writing", "bookId": "<id>" }
// SSE start
{ "event": "write:start",    "data": { "bookId": "<id>" } }
// SSE success
{ "event": "write:complete", "data": { "bookId": "<id>", "chapterNumber": 3, "status": "ready-for-review", "title": "...", "wordCount": 2800 } }
// SSE error
{ "event": "write:error",    "data": { "bookId": "<id>", "error": "..." } }
```

**POST `/api/books/:id/draft`**（可控流程第一步：预览 plan）
```json
// 请求体（可选）
{ "context": "本章把注意力拉回师徒矛盾", "wordCount": 2500 }
// 响应
{ "status": "drafting", "bookId": "<id>" }
// SSE start
{ "event": "draft:start",    "data": { "bookId": "<id>" } }
// SSE success
{ "event": "draft:complete", "data": { "bookId": "<id>", "chapterNumber": 3, "title": "...", "wordCount": 2500 } }
// SSE error
{ "event": "draft:error",    "data": { "bookId": "<id>", "error": "..." } }
```

### 10.3 Copy-paste 级回归步骤

```bash
# 0. 启动 Studio
inkos studio

# 1. 确认书籍存在，记录下一章章号（nextChapter 即将写入的章节号）
curl -s http://localhost:8765/api/books/<BOOK_ID> | jq .nextChapter

# 2. 预览 plan（填干预 + 草稿）
curl -s -X POST http://localhost:8765/api/books/<BOOK_ID>/draft \
  -H "Content-Type: application/json" \
  -d '{"context":"本章聚焦师债主线冲突"}' | jq .
# → {"status":"drafting","bookId":"<BOOK_ID>"}

# 3. 订阅 SSE 事件（另一个终端）
curl -N http://localhost:8765/api/events
# → 等待 draft:start ... draft:complete

# 4. 写下一章
curl -s -X POST http://localhost:8765/api/books/<BOOK_ID>/write-next \
  -H "Content-Type: application/json" \
  -d '{}' | jq .
# → {"status":"writing","bookId":"<BOOK_ID>"}

# 5. 等待 SSE write:start ... write:complete

# 6. 验证新章节
curl -s http://localhost:8765/api/books/<BOOK_ID> | jq '.chapters[-1]'
# → {"number":N,"status":"ready-for-review",...}
```

## 12. 运行中心端到端验收与运维

### 12.0 守护进程模式与入口说明（先读）

#### 12.0.1 守护进程两种模式

| 模式 | API 入口 | 推荐场景 |
|------|----------|----------|
| 默认模式（managed-default） | `POST /api/daemon/start`（`{ "default": true }` 或空 body） | 新手一键启动，按项目默认调度配置运行 |
| 高级模式（custom-plan） | 先 `POST /api/daemon/plan` 创建计划，再 `POST /api/daemon/start` 传 `planId` | 高级用户需要控制书籍范围与章节预算 |

#### 12.0.2 Studio daemon vs CLI daemon

| 对比项 | Studio daemon（运行中心） | CLI daemon（命令行） |
|--------|----------------------------|----------------------|
| 启停方式 | `/api/daemon/start|pause|resume|stop`（UI 可点击） | `inkos up` / `inkos down` |
| 可观测性 | `/api/daemon/session` + `/api/runtime/events` + UI 状态面板 | 终端输出 + `inkos.log` + `inkos.pid` |
| 推荐使用者 | 需要可视化和验收记录的团队/QA | 纯脚本、SSH、无人值守命令行场景 |

### 12.0.3 端到端验收清单（含 curl + UI 路径）

- [ ] **默认模式 curl 路径**
  - 执行 `POST /api/daemon/start`（`{"default":true}`）
  - 校验返回 `{ "ok": true, "running": true, "mode": "managed-default" }`
- [ ] **高级模式 curl 路径**
  - 执行 `POST /api/daemon/plan`，body 至少包含 `plan.mode / plan.bookScope / perBookChapterCap / globalChapterCap`
  - 使用响应 `planId` 调用 `POST /api/daemon/start`
  - 校验返回 `{ "ok": true, "running": true, "planId": "...", "mode": "custom-plan" }`
- [ ] **UI 手测路径（默认模式）**
  - 进入 Studio 首页 → 左侧导航「运行中心」
  - 在「运行模式」选择「默认模式」→ 点击「启动」
  - 会话状态显示 `running`（或「运行中」），事件流出现 `daemon:started`
- [ ] **UI 手测路径（高级模式）**
  - 进入「运行中心」→ 选择「高级模式」
  - 填写目标书籍与预算参数（每本章节上限、全局章节上限等）→ 点击「启动」
  - 事件流出现 `daemon:started`，且事件 data 中 `mode` 为 `custom-plan`
- [ ] **暂停 / 恢复 / 停止**
  - 依次点击「暂停守护进程」「恢复守护进程」「停止」
  - 事件流依次可见 `daemon:paused`、`daemon:resumed`、`daemon:stopped`
- [ ] **历史事件可追溯**
  - 浏览器刷新运行中心页面后，`GET /api/runtime/events` 仍能查到刷新前事件

### 12.1 标准链路 A：启动守护 → 状态变化与事件可见

```bash
# 0) 启动 Studio（运行中心在 Web UI 内）
INKOS_PROJECT_ROOT=/path/to/your-project pnpm --filter @actalk/inkos-studio dev

# 1) 启动守护
curl -s -X POST http://localhost:4569/api/daemon/start | jq .

# 2) 查看运行中心概览状态
curl -s http://localhost:4569/api/runtime/status | jq .

# 3) 查询运行中心事件流（守护来源）
curl -s "http://localhost:4569/api/runtime/events?source=daemon&limit=20" | jq .
```

预期：
- `POST /api/daemon/start` 返回 `{ "ok": true, "running": true }`
- `GET /api/runtime/status` 中 `daemonRunning=true`、`eventCount` 递增
- `GET /api/runtime/events` 可见 `daemon:started`（停止后还会有 `daemon:stopped`）

### 12.2 标准链路 B：触发精修 → 提交/执行/完成(或失败)全过程可见

```bash
BOOK_ID="<your-book-id>"
CHAPTER_NO=3

# 1) 触发精修
curl -s -X POST "http://localhost:4569/api/books/$BOOK_ID/revise/$CHAPTER_NO" \
  -H "Content-Type: application/json" \
  -d '{"mode":"rewrite","brief":"保持角色动机一致，修复节奏断裂"}' | jq .

# 2) 查询该书在运行中心的历史事件
curl -s "http://localhost:4569/api/runtime/events?bookId=$BOOK_ID&limit=100" | jq .
```

预期（成功路径）：
- 事件包含 `revise:start` → `revise:success`
- payload 含 `action`、`chapterNumber`、`briefUsed`、`bookId`

预期（失败路径）：
- 事件包含 `revise:fail`（或同类 `*:fail` 语义失败事件）
- 运行中心中失败事件可被识别为错误级别并用于筛选

### 12.3 标准链路 C：刷新页面 → 历史事件可查询

说明：浏览器刷新会重建 SSE 连接，但不会清空运行中心后端内存事件缓冲。

```bash
# 刷新前查询一次
curl -s "http://localhost:4569/api/runtime/events?limit=20" | jq '.total'

# 浏览器刷新运行中心页面后，再次查询
curl -s "http://localhost:4569/api/runtime/events?limit=20" | jq '.total'
```

预期：
- 两次查询均成功，刷新后仍可取到刷新前已产生的历史事件
- `total` 不会因前端刷新归零（除非主动调用清空接口）

### 12.4 故障排查（运行中心）

| 症状 | 诊断步骤 | 处理建议 |
|------|----------|----------|
| SSE 断线（运行中心不再滚动） | 1) 打开浏览器 Network，确认 `/api/events` 是否断开；2) `curl -N http://localhost:4569/api/events` 检查是否持续收到事件；3) `curl -s /api/runtime/status` 查看 `sseClientCount` | 重启 Studio API 进程；确认反向代理未切断长连接；排除本机网络/防火墙对 EventSource 的影响 |
| 锁文件导致守护无法启动 | 1) `POST /api/daemon/start` 查看报错；2) 检查项目目录下运行锁文件（如残留 `.lock`）；3) 核对是否已有旧守护进程占用 | 结束旧进程后清理残留锁文件，再次启动；避免多实例同时操作同一项目目录 |
| 空日志（界面看不到历史） | 1) 先查 `GET /api/runtime/events`（运行中心主数据源）；2) 再查 `GET /api/logs`（兼容接口，可能为空）；3) 核对是否刚执行过 `/api/runtime/clear` | 优先以运行中心事件接口判定系统状态；确认未误清空；若仅旧日志为空属于兼容路径现象 |

## 13. 全章节操作可观测性验收（含重写/反检测）

### 13.1 标准链路（提交建议 → runId → 事件流 → 最终状态 → diff）

| 步骤 | 接口 | 验收点（字段名与 API 一致） |
|------|------|-----------------------------|
| 1. 提交建议 | `POST /api/books/:id/revise/:chapter` | 返回 `status`、`runId`、`bookId`、`chapter`、`mode`、`appliedBrief` |
| 2. 记录 runId | 第 1 步响应体 | `runId` 为后续查询唯一标识 |
| 3. 追踪事件流 | `GET /api/books/:id/chapter-runs/:runId/events` | 返回 `{ runId, events }`；事件按 `start -> success/fail`，字段含 `type`、`status`、`timestamp` |
| 4. 核对最终状态 | `GET /api/books/:id/chapter-runs/:runId` | 核对 `status`（`running/succeeded/failed`）、`decision`（`applied/unchanged/failed`）、`unchangedReason`、`error` |
| 5. diff 回看 | `GET /api/books/:id/chapter-runs/:runId/diff` | 返回 `beforeContent`、`afterContent`、`briefTrace`，并保留 `runId/status/decision/unchangedReason` |

> 备注：`mode` 可直接使用 `"rewrite"` 或 `"anti-detect"`；两者都写入 chapter run，并可复用相同验收链路。

### 13.2 失败分类（含用户可见提示）

| 失败分类 | 用户可见提示（当前实现） | 诊断接口/信号 | 处理动作 |
|----------|--------------------------|---------------|----------|
| 模型空返回（建议为空/占位） | `POST /api/books/:id/next-plan` 返回 `error.code = "PLAN_LOW_CONFIDENCE"`（HTTP 409） | `plan:fail` 事件 + 接口错误体 | 调整 `planInput` 后重试，或切到手动规划 |
| 质量门槛回滚（重写链路失败） | 运行中心出现 `rewrite:fail`，`message/error` 为失败原因 | `GET /api/runtime/events?bookId=<id>` + `GET /api/books/:id/chapter-runs/:runId` (`status = "failed"`) | 回看失败 run 的 `error`，修正 brief 后重试 |
| 事件漏接（前端断流） | 页面不滚动但操作实际已完成 | `GET /api/books/:id/chapter-runs/:runId/events` 仍有完整历史 | 以 run 事件接口回放；必要时重连 `/api/events` |
| 落盘失败（写文件失败） | chapter run `status = "failed"` 且 `error` 含文件写入异常 | `GET /api/books/:id/chapter-runs/:runId` + `*:fail` 事件 | 检查磁盘空间/目录可写性后重试 |
| 权限/路径错误 | 非法 bookId 返回 `error.code = "INVALID_BOOK_ID"`（HTTP 400）或对应 4xx/5xx 错误 | 直接看接口响应体 | 修正 bookId/路径后重试，避免路径穿越输入 |

### 13.3 一键验收命令（可复制）

```bash
BOOK_ID="demo-book" CHAPTER=3 BASE_URL="http://localhost:4569" bash -lc '
set -euo pipefail
for MODE in rewrite anti-detect; do
  echo "== submit mode=$MODE =="
  RESP=$(curl -s -X POST "$BASE_URL/api/books/$BOOK_ID/revise/$CHAPTER" \
    -H "Content-Type: application/json" \
    -d "{\"mode\":\"$MODE\",\"brief\":\"验收-$MODE\"}")
  echo "$RESP" | jq .
  RUN_ID=$(echo "$RESP" | jq -r ".runId")
  test -n "$RUN_ID" && test "$RUN_ID" != "null"

  for _ in $(seq 1 30); do
    STATUS=$(curl -s "$BASE_URL/api/books/$BOOK_ID/chapter-runs/$RUN_ID" | jq -r ".status")
    [ "$STATUS" != "running" ] && break
    sleep 1
  done

  echo "== events runId=$RUN_ID =="
  curl -s "$BASE_URL/api/books/$BOOK_ID/chapter-runs/$RUN_ID/events" | jq .
  echo "== final runId=$RUN_ID =="
  curl -s "$BASE_URL/api/books/$BOOK_ID/chapter-runs/$RUN_ID" | jq .
  echo "== diff runId=$RUN_ID =="
  curl -s "$BASE_URL/api/books/$BOOK_ID/chapter-runs/$RUN_ID/diff" | jq "{runId,status,decision,unchangedReason,beforeContent,afterContent,briefTrace}"
done
'
```
