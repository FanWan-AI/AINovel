# 运维部署与可观测性

## 1. 部署形态

当前建议维持：
- Studio：Vite + Hono Node Server
- Core/CLI：Node 运行时
- 配置：`inkos.json` + `.env`

## 2. 环境分层

- `dev`：本地开发环境
- `staging`：集成测试环境
- `prod`：生产环境（或用户本地生产运行）

配置管理原则：
- API Key 仅放 `.env`
- `inkos.json` 不存密钥

## 3. 运行与启动

建议标准命令：
- `pnpm dev`（开发）
- `pnpm build && pnpm -r test`（构建与测试）

Studio 默认端口：`4567`

## 4. 可观测性设计

### 4.1 日志

日志最小字段：
- `timestamp`
- `level`
- `traceId`
- `bookId`
- `stage`
- `message`
- `errorCode`

日志级别约定：
- info：流程进展
- warn：可恢复异常
- error：失败或不可恢复错误

### 4.2 指标

核心指标：
- 创建成功率
- 创建耗时（P50/P95）
- normalize 失败率
- 章节生成失败率
- 连续性告警平均数

### 4.3 事件

关键事件通过 SSE 推送与日志同时记录，便于前后排障一致。

## 5. 健康检查

建议新增 `/api/healthz`（或增强现有健康接口）：
- API 可用
- 项目配置可读
- LLM 配置有效性检查（非强制调用模型）

## 6. 故障处理 Runbook

故障 A：创建长期卡在 `creating`
- 检查 `bookCreateStatus` map 与日志
- 检查模型服务连通
- 查看目标目录写权限

故障 B：403/401 模型拒绝
- 检查 baseUrl / model / apiKey 三元组
- 执行 `inkos doctor` 验证配置连通

故障 C：章节连贯性异常增多
- 检查 brief 是否缺失关键字段
- 检查 `current_focus` 与 plan 输出是否漂移

## 7. 备份与恢复

建议备份目录：
- `books/`
- `inkos.json`
- `.env`（单独加密管理）

恢复流程：
1. 恢复 `books` 与配置。
2. 执行一致性检查脚本（后续可补）。
3. 从最近章节继续生成。

## 8. 容量与成本建议

- normalize 使用轻量模型优先，减少高成本推理。
- 章节写作使用主力模型，审稿可用强推理模型。
- 对长输入进行长度分段与摘要化，降低 OOM 风险。
