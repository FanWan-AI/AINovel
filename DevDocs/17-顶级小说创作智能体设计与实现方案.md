# 顶级小说创作智能体设计与实现方案

## 1. 文档目标

本文档回答三个问题：

1. 当前 AI 助手已经实现了什么，优点与不足是什么。
2. 面向产品最终形态，如何把 AI 助手升级为“仅靠聊天即可写好一本小说”的核心智能体。
3. 如何基于现有代码最小返工、分阶段落地，并确保可观测、可评估、可持续迭代。

适用范围：InkOS Studio 的 assistant 主页面、ChatPanel、Studio API、Core Pipeline。

---

## 2. 当前实现能力审计（基于现有代码）

### 2.1 已实现能力（事实清单）

1. 已有独立助手页面框架（ContextBar + MessageStream + Composer）。
2. 已有书籍范围选择（single/multi/all-active）与阻断提示。
3. 已有书籍级动作识别（write-next、audit）与参数确认卡片。
4. 已有右侧 ChatPanel 的动作识别与 API 映射（write-next/audit/market-radar）。
5. 已有 SSE 事件反馈与阶段状态文本回显。
6. 已有后端原子动作接口：write-next、audit、revise、rewrite、resync、radar。
7. 已有章节运行记录与 diff/approve/delete 能力（chapter-runs ledger）。
8. 已有写作偏好治理接口与设置页承接（writing-governance）。
9. 已有 core pipeline 的多智能体资产：planner/composer/writer/reviser/continuity/radar 等。

### 2.2 当前方案的优点

1. 可运行闭环已具备基础形态：聊天可触发真实动作。
2. 风险控制初步到位：书籍范围阻断、确认卡、章节 run ledger。
3. 后端能力储备充足：核心创作链路与审计修订链路已存在。
4. 工程可演进性较好：助手页、设置页、runtime 事件中心已形成骨架。

### 2.3 当前不足与根因

1. 交互层存在“双助手入口割裂”：AssistantView 和 ChatPanel 语义不一致。
2. AssistantView 仍以模拟回复为主，缺少真实工具编排执行器。
3. 意图理解能力弱：仅靠 regex，无法处理复杂复合任务与追问澄清。
4. 缺少长期记忆编排：会话上下文、书籍状态、用户偏好未统一决策。
5. 缺少质量闭环评分：结果不是“可量化好坏”，难以自动改进。
6. 缺少可解释性与可追踪：用户看得到结果，但看不清依据与决策链。
7. 缺少“一键继续下一步”的增长飞轮动作设计。

结论：当前 AI 助手是“可用原型”，不是“顶级创作智能体产品”。

## 3. 用户痛点与最终形态要求

### 3.1 用户核心痛点

1. 不知道先做什么：开书、推进、审校、修订路径不清楚。
2. 不知道为什么改：审稿建议和修改动作缺乏明确因果链。
3. 不敢全自动：担心改崩剧情、人物设定、章节节奏。
4. 成本焦虑：多轮对话和重复操作造成时间与 token 浪费。
5. 无法“全局掌控一本书”：章节质量、伏笔兑现、风格一致性难量化。

### 3.2 最终形态目标

1. 用户通过聊天即可完成一本小说的增删改查。
2. 助手可自动执行“写作-审计-修复-复审-发布候选”全链路。
3. 用户可选择全自动、半自动、人工审批三种模式。
4. 每一步都有可解释证据与可回滚机制。
5. 结果以质量评分和风险等级呈现，而不是仅文本输出。

## 4. 主流 Agent 设计思想吸收

### 4.1 Anthropic（Building Effective Agents）

可直接吸收：

1. 先工作流再自治：优先简单可控，再提升自治度。
2. 五类高价值模式：Routing、Parallelization、Orchestrator-Workers、Evaluator-Optimizer、Agent Loop。
3. 强调 ACI（Agent-Computer Interface）：工具定义清晰比“花哨框架”更关键。
4. 强调停机条件和人工检查点，避免无界循环。

### 4.2 OpenHands

可直接吸收：

1. 产品分层：SDK + CLI + GUI + Cloud，降低不同用户门槛。
2. 任务执行的可观测与评测基础设施优先建设。
3. 协作能力（分享、权限、预算）作为后期放大器。

### 4.3 AutoGen

可直接吸收：

1. 多 Agent Team 结构与 Selector 模式。
2. GraphFlow（图编排）适合复杂创作流水线。
3. Memory 与 Logging 作为内建一等公民。

### 4.4 OpenClaw

可直接吸收：

1. local-first gateway：本地控制面 + 多通道接入。
2. skills registry：能力插件化、可版本化、可治理。
3. 安全默认策略（配对、白名单、沙箱）提升可控性。
4. Operator 命令体系（status/reset/trace）提升可运维性。

### 4.5 Harmess 说明

当前公开检索未找到明确的 “harmess agent” 权威项目。更可能是 Harness 或其他近似拼写项目。

处理策略：

1. 方法论上吸收 Harness 的“生产级治理、可观测、发布门槛”思想。
2. 不绑定不确定品牌名，避免架构决策建立在错误事实之上。

### 4.6 OpenClaw + Harness 融合落地原则

1. OpenClaw 侧重点用于“可用与易用”：单入口、命令面、技能生态、会话控制。
2. Harness 侧重点用于“可控与可上线”：策略门禁、预算控制、灰度发布、复盘闭环。
3. NovelOS 的核心策略是“先像 OpenClaw 一样好用，再像 Harness 一样可规模治理”。

## 5. 目标方案：InkOS NovelOS Agent

### 5.1 一句话定义

从“聊天助手”升级为“小说创作操作系统”：用户只描述目标，智能体自动拆解、执行、评估、修复，并在关键节点征求用户意图。

### 5.2 北极星能力

1. Goal-to-Book：从目标到可发布书稿的端到端闭环。
2. Explainable-by-default：每次改动都能追溯原因、证据、影响。
3. Human-in-control：任何高风险动作默认可审批、可回滚。
4. Self-improving：失败与低分结果自动触发修复与策略更新。

## 6. 顶层架构设计

### 6.1 六层架构

1. 对话交互层（Conversational UX）
2. 意图与任务图层（Intent Router + Task Graph Planner）
3. 执行编排层（Orchestrator + Worker Agents）
4. 质量与审查层（Evaluator-Optimizer + Guardrails）
5. 记忆与知识层（Session/Book/World Memory）
6. 观测与治理层（Tracing/Metrics/Policy/Release Gates）

### 6.2 Agent Team 角色

1. Conductor（总调度）：拆解任务图、分配 worker、管理停机条件。
2. Story Architect：长线规划、卷章目标、冲突结构。
3. Chapter Writer：生成章节草稿与正式稿。
4. Quality Auditor：连续性、逻辑、节奏、风格、AI 痕迹审查。
5. Revision Surgeon：定点修复、重写、反检测改写。
6. World Consistency Keeper：人物/设定/伏笔一致性守门。
7. Market Strategist：题材趋势、节奏与商业化建议。
8. Release Editor：综合评分、给出“可发布候选”。

### 6.3 核心模式组合

1. Routing：把用户需求路由到“创作/审校/运维/知识问答”。
2. Orchestrator-Workers：复杂任务拆成并行子任务。
3. Evaluator-Optimizer：每轮生成后自动审查并迭代。
4. Parallelization Voting：关键章节可多候选投票选优。
5. Human Checkpoints：高风险改动必须经过确认。

### 6.4 ACI（Agent-Computer Interface）标准

1. 每个工具必须有：用途说明、输入约束、失败码、示例。
2. 每次工具调用必须输出：`toolName`、`taskId`、`stepId`、`durationMs`、`status`。
3. 工具参数优先用自然可读字段，避免让模型维护复杂格式计数。
4. 同类工具必须有清晰边界，禁止“语义重叠工具”并存。
5. 编排器必须支持 tool fallback（首选失败时降级到替代工具）。

## 7. 用户体验设计（比当前更简单）

### 7.1 统一单入口

1. 废弃语义割裂：AssistantView 成为唯一主入口。
2. ChatPanel 转为“轻量快捷窗”，仅做跳转与状态提示，不承担主编排。

### 7.2 对话体验原则

1. 先给结果，再给理由，再给下一步按钮。
2. 所有回复都包含“执行对象 + 风险等级 + 预计耗时”。
3. 所有失败都包含“一键重试 / 改参数 / 降级执行”。

### 7.3 一键飞轮

每次任务完成后必须给出下一步卡片：

1. 继续写下一章
2. 对本章做深度审计
3. 自动修复并复审
4. 更新全书一致性报告

### 7.4 命令面（Operator Surface）

参考 OpenClaw 的 operator 设计，助手支持显式命令以降低学习成本：

1. `/goal <描述>`：创建目标任务。
2. `/status`：查看当前任务状态与进度。
3. `/pause` 与 `/resume`：暂停/恢复自动流程。
4. `/approve <step|task>`：审批高风险步骤。
5. `/rollback <runId>`：回滚指定运行结果。
6. `/trace on|off`：开关详细执行轨迹。
7. `/budget`：查看 token 与成本预算消耗。

### 7.5 新手即用模板（Prompt Shortcuts）

为降低首次使用门槛，助手内置模板按钮：

1. “从一句话梗概生成 3 卷 30 章结构”。
2. “按当前设定写下一章并自审一次”。
3. “审计最近三章并给最小修复方案”。
4. “生成本周更新计划（字数/节奏/伏笔）”。

模板执行规则：

1. 先生成可预览计划，再执行。
2. 高风险模板默认 L1（逐步确认）。
3. 每次模板执行后自动产出下一步建议卡。

## 8. 能力模型：聊天即 CRUD + 审核 + 自动改进

### 8.1 增（Create）

1. 新建书、生成蓝图、生成章节计划。
2. 支持全自动连载模式和用户逐章确认模式。

### 8.2 删（Delete）

1. 删除章节、删除候选 run、删除无效设定。
2. 删除动作均支持预览影响和软删除恢复。

### 8.3 改（Update）

1. 章节改写（spot-fix/polish/rewrite/rework/anti-detect）。
2. 世界观、人物弧线、伏笔状态自动联动修正。

### 8.4 查（Read）

1. 按书、卷、章、角色、冲突、伏笔查询。
2. 提供证据视图：引用章节片段与状态文件来源。

### 8.5 全面审核

1. 章节级评分：逻辑、节奏、可读性、风格一致性、AI 痕迹。
2. 全书级评分：主线推进、角色成长、伏笔兑现率、重复度。

### 8.6 自动改进

1. 低分自动触发修复环（Evaluator-Optimizer）。
2. 用户可设置自动阈值与最大迭代次数。
3. 超过阈值仍未达标时自动转人工审批。

### 8.7 自动驾驶等级（Autopilot Levels）

1. L0 手动：只给建议，不自动执行。
2. L1 辅助：每步确认后执行。
3. L2 半自动：低风险自动执行，高风险确认。
4. L3 全自动：在预算与策略约束下全链路自动执行。

默认策略：新用户 L1，活跃用户 L2，仅白名单用户可启用 L3。

## 9. 关键数据模型

### 9.1 TaskGraph

1. taskId
2. parentTaskId
3. intentType
4. nodes（步骤）
5. edges（依赖）
6. status
7. checkpoints

### 9.2 QualityReport

1. scope（chapter/book）
2. scoreBreakdown
3. blockingIssues
4. suggestedFixes
5. confidence

### 9.3 Memory Layers

1. Session Memory：当前会话短期上下文。
2. Book Memory：本书中长期状态与关键事实。
3. User Preference Memory：用户偏好、语气、风险偏好。
4. Market Memory：外部趋势与题材信号。

### 9.4 Skill Registry（技能注册表）

参考 OpenClaw/ClawHub 思路，NovelOS 引入技能注册层：

1. Built-in Skills：平台内置动作（write/audit/revise/radar）。
2. Project Skills：项目级自定义能力（题材规则、风格插件）。
3. Trusted Skills：受信任外部技能（需签名与权限声明）。

每个技能必须声明：

1. `name/version`。
2. `requiredEnv` 与 `requiredBins`。
3. `riskLevel`（low/medium/high）。
4. `allowedScopes`（single/multi/all-active）。
5. `rollbackSupport`（是否支持回滚）。

## 11. 分阶段落地路线图

### Phase A（2 周）：可用化

1. AssistantView 接入真实执行，不再使用模拟回复。
2. 统一 AssistantView 与 ChatPanel 动作语义。
3. 补齐参数确认、失败恢复、进度可视化。

2. 接入 Evaluator-Optimizer 双环。

1. 全书质量报告与发布候选判定。
2. 策略配置中心（自动化阈值、审批策略、预算）。
3. 观测看板（成功率、失败率、迭代成本、留存指标）。

## 12. 可观测与评估体系

### 12.1 核心指标

1. 首次任务成功率
2. 会话到执行转化率
3. 自动修复成功率
4. 人工介入率
5. 章节平均质量分
6. 每章平均 token 成本
7. 7 日留存与周活跃创作者占比

### 12.2 质量门槛（示例）

1. 章节质量分 < 70：禁止自动发布
2. 连续两轮修复无提升：转人工确认
3. 全书一致性评分低于阈值：触发全局修复任务

### 12.3 易用性与智能性指标

1. 首次上手（First 10 min）完成首章写作比例 >= 70%。
2. 平均每个任务的人工点击次数 <= 3。
3. 用户对“建议可解释性”满意度 >= 4.3/5。
4. 自动优化后平均分提升 >= 8 分。
5. 无效自动执行回滚率 <= 3%。

## 13. 风险与防护

1. 幻觉与错误修改：必须附证据引用与一致性校验。
2. 自动改坏剧情：高风险动作默认确认 + 一键回滚。
3. 成本失控：任务步数、token、并行数设硬上限。
4. 模型不稳定：模型路由与降级策略（强模型审定、弱模型草拟）。
5. 安全风险：敏感工具白名单 + 分级权限 + 审计日志。

补充防护机制：

1. 策略引擎（Policy-as-Code）：在执行前校验风险、预算、权限。
2. 预算守卫（Budget Guard）：超预算自动降级到 L1 或暂停。
3. 提示注入防护：用户输入与工具参数隔离，敏感字段不拼接入 prompt。
4. 发布门禁：未通过质量门槛和审查门槛的结果不可进入发布候选。

## 15. 最终结论

1. 你们当前不是“从零开始”，而是“核心引擎强、助手层偏弱”。
2. 只要完成编排层与体验层升级，就能从“可用”跃迁到“可爆发增长”。
3. 这套方案的关键不是增加更多模型，而是：统一入口、任务图编排、质量闭环、可观测治理。

该方案比“单轮聊天助手”更先进之处在于：

1. 能真正完成一本书，而不是只写几段文本。
2. 能自动审核并持续改进，而不是靠用户反复提示。
3. 能在保证可控和可回滚的前提下实现高自动化。

## 17. 未完成目标细化设计

### 17.1 U1 — TaskGraph 编排引擎

#### 数据模型

```typescript
interface TaskGraph {
  taskId: string;              // uuid
  parentTaskId?: string;       // 支持嵌套
  intentType: 'create' | 'audit' | 'revise' | 'goal-to-book' | 'composite';
  nodes: TaskNode[];
  edges: TaskEdge[];           // {from, to, condition?}
  status: 'planning' | 'running' | 'paused' | 'succeeded' | 'failed';
  checkpoints: Checkpoint[];   // 人工审批点
  createdAt: number;
  updatedAt: number;
}

interface TaskNode {
  nodeId: string;
  skillName: string;           // 来自 Skill Registry
  params: Record<string, unknown>;
  status: 'pending' | 'running' | 'succeeded' | 'failed' | 'skipped';
  retryCount: number;
  maxRetries: number;
  result?: unknown;
  durationMs?: number;
}

interface TaskEdge {
  from: string;    // nodeId
  to: string;      // nodeId
  condition?: 'on-success' | 'on-failure' | 'always';
}

interface Checkpoint {
  nodeId: string;
  requiredApproval: boolean;
  approvedAt?: number;
  approvedBy?: string;
}
```

#### Conductor 调度器

- 位置：`packages/studio/src/api/services/assistant-conductor.ts`
- 职责：
  1. 接收 TaskGraph，拓扑排序确定执行顺序
  2. 管理并行分支（无依赖节点可同时执行）
  3. 遇到 Checkpoint 时暂停并通知前端等待审批
  4. 失败节点根据 `maxRetries` 重试，超限标记失败
  5. 全图完成后汇总结果、生成执行报告
- 核心方法：`runGraph(graph: TaskGraph): AsyncGenerator<TaskEvent>`
- 前端面板：AssistantView 新增 `TaskGraphPanel` 以可视化方式展示 DAG 节点状态

#### API 变更

- `POST /api/assistant/plan` 返回值增加 `graph: TaskGraph` 字段
- `POST /api/assistant/execute` 接受 `taskId`，由 Conductor 驱动执行
- 新增 `POST /api/assistant/tasks/:taskId/approve/:nodeId` 用于审批 Checkpoint

### 17.2 U2 — 多候选并行投票

#### 设计

- 场景：关键章节（如高潮、转折点）可生成 2-3 个候选版本
- 触发条件：节点配置 `parallelCandidates: number`（默认 1，即不并行）
- 投票策略：
  1. 自动投票：对每个候选运行 `/api/assistant/evaluate`，选最高分
  2. 人工投票：L1/L2 模式下展示候选对比卡，用户点选
- 实现位置：Conductor 的 `executeNode()` 方法内，检测 `parallelCandidates > 1` 时 fork
- 前端组件：`CandidateComparisonCard` 展示候选差异与评分

### 17.3 U3 — Autopilot L2/L3

#### 设计

L2（半自动）：
- 低风险动作（riskLevel=low）自动执行，跳过确认
- 中风险（medium）展示确认卡但自动倒计时（30s），用户不点击则自动执行
- 高风险（high）必须人工确认

L3（全自动）：
- 所有动作自动执行，仅在预算耗尽或连续失败 2 次时暂停
- 需白名单启用，策略配置中心配置

#### 实现要点

- `assistant-policy-service.ts` 增加 `autopilotLevel` 参数，根据等级决定是否插入 Checkpoint
- `assistant-conductor.ts` 在节点执行前查询策略，动态决定跳过或暂停
- 前端 SettingsView 新增 autopilot 等级选择控件
- 安全约束：L3 模式下仍受 Budget Guard 硬限制

### 17.4 U4 — Memory 分层体系

### 17.5 U5 — Goal-to-Book 端到端管线
